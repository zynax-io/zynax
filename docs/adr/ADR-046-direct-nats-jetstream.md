# ADR-046: Direct NATS JetStream via a shared client library — the EventBusService gRPC facade is retired

**Status:** Accepted  **Date:** 2026-07-03
**Related:** ADR-022 (**amends** — reverses its Option 1: producers/subscribers no longer call `EventBusService.Publish`/`Subscribe`; they use JetStream directly through a shared client that preserves the subject/DLQ/terminal-close conventions, and lifts its "no NATS client in agents" pre-commitment for the optional SDK path), ADR-001 (carves a **narrow exception** — eventing is fire-and-forget pub/sub over JetStream, not request/response inter-service RPC, and stays gRPC-free by design), ADR-013 (**upheld** — the SDK stays optional and adapters stay gRPC-only; adding a `nats-py` eventing path to `agents/sdk` is consistent with ADR-013, not an exception to it), ADR-008 (upheld — durability stays in JetStream; no service DB), ADR-006 (module layout — the shared client is a new versioned `libs/` module wired via `go.work`, like `libs/zynaxconfig` / `libs/zynaxobs`), ADR-017 (upheld — the new module complies with the GOWORK=off contract-test regime), ADR-020 (**upheld** — the JetStream link authenticates with the same cert-manager-issued per-service certs via NATS `verify_and_map`; no second secret system, no ADR-020 Option-C shared-secret regression), ADR-040 (§1 delegation — direct JetStream removes a bespoke Zynax service instead of adding one), ADR-041 (the mandated NATS TLS provisioning now lands on the kind first-run path — kept one-command and zero-secret via cert-manager auto-issue). Resolves EPIC #1576 (M8.H); canvas `docs/spdd/1576-direct-nats-jetstream/canvas.md`. Planned sibling M8 ADRs (co-accepted in this batch): ADR-043 (#1573), ADR-044 (#1574 — edge rate-limiting, M8.F), ADR-045 (#1575).

---

## Context

ADR-022 chose **Option 1** — a full gRPC `EventBusService` fronting NATS JetStream — over a shared-library approach (its Option 2). Two years of usage let us test that decision against reality, and the load-bearing premises did not hold:

- **The facade enforces zero policy.** ADR-022 §Rationale 3 justified the service as the "natural chokepoint" for topic authorization, namespace isolation, and rate limits ("planned for M7"). Grepping `services/event-bus/internal/` for authorization or rate-limit logic finds none — only a passthrough `Namespace` field and JetStream retention-policy names (`LimitsPolicy` / `WorkQueuePolicy`); no authz, quota, or rate enforcement exists. The chokepoint was never built. Removing the facade removes **no security control that exists today**.
- **All the real logic is convention, not service.** `services/event-bus/internal/api/handler.go` (~168 LOC) only validates non-empty id/source/type and maps proto↔domain. Everything non-trivial lives in `services/event-bus/internal/infrastructure/nats.go` (~543 LOC): the depth-4 subject→stream derivation with the #1149 pairwise-disjoint-filter rule (guards NATS err 10065), the `DLQ_<src>` / `zynax.dlq.<prefix>.dead` WorkQueue machinery, `RetryBackoff`+`MaxDeliver=5`, durable-consumer name sanitizing, the `MatchesGlob` matcher, and the workflow_id-scope + terminal-event stream-close (`domain.IsTerminalEventType` in `services/event-bus/internal/domain/event.go`). None of this needs a network hop — it is library code wearing a Deployment.
- **The transport-security claim is aspirational, not inherited.** The mTLS in event-bus's `tlscreds.go` protects its *inbound gRPC*, not the NATS link. The community `nats/nats` subchart (`helm/charts/nats/`) is deployed with **no accounts, no creds, no NATS-level TLS**; event-bus dials `nats://zynax-nats:4222` in plaintext (`helm/zynax-event-bus/values.yaml`). Only NetworkPolicy bounds NATS. So per-service NATS credentials + link TLS are **net-new work in either design** — the facade never provided them.
- **The client surface is tiny.** Three best-effort publishers — engine-adapter (`internal/infrastructure/activities.go`), task-broker (`internal/infrastructure/event_publisher.go`), api-gateway (`internal/infrastructure/clients.go`) — and exactly one subscriber: api-gateway's Subscribe→REST bridge (`clients.go` `SubscribeWorkflowEvents`, backing `WatchWorkflowLogs` in `internal/domain/apply.go`). No Python code publishes or subscribes today (`agents/` has zero `EventBusService`/`event_bus_pb2` hits).

ADR-022 called Option 1 a one-way door because the proto + BDD contracts were committed first. That framing is now inverted: the *contract of record* for events is the **AsyncAPI spec** (`spec/asyncapi/zynax-events.yaml`, with the CloudEvents envelope and `zynaxschemarev` versioning), not the gRPC service. The facade is an implementation detail behind that contract, and ADR-040 §1 explicitly rejects standing up a Zynax service where an infrastructure primitive (JetStream) already does the durable work. This ADR reverses ADR-022 and confronts its rationale point by point.

---

## Decision

**We will retire the `EventBusService` gRPC facade and have publishers and subscribers use NATS JetStream directly through a thin, shared client library that preserves the subject conventions, DLQ behaviour, and terminal-close semantics verbatim. The AsyncAPI spec remains the single contract of record. Deprecate in M8, remove in M9.**

1. **Shared Go client = `libs/zynaxevents` (new module).** A new `libs/` module (own `go.mod`, consumed via per-service `replace … => ../../libs/zynaxevents` + `go.work`, exactly like `libs/zynaxconfig` / `libs/zynaxobs`, per the ADR-006 monorepo `go.work` layout; the new module complies with the ADR-017 GOWORK=off contract-test regime). It moves the `nats.go` conventions **verbatim**: depth-4 stream derivation + #1149 disjoint-filter rule, `DLQ_<src>` / `zynax.dlq.<prefix>.dead` / WorkQueuePolicy, `RetryBackoff` + `MaxDeliver=5`, durable-name sanitizing, `MatchesGlob`, CloudEvents JSON + `ce-*` headers, and `zynaxobs` trace inject/extract. Golden-event byte-compat and DLQ/durable-consumer BDD are acceptance gates.
2. **The contract is AsyncAPI + the shared library — not gRPC.** `spec/asyncapi/zynax-events.yaml` stays the source of truth (channels, envelope, `zynaxschemarev`); the reference stream list in `helm/charts/nats/values.yaml` stays as documentation, now realised by the library rather than enforced by a service. `protos/zynax/v1/event_bus.proto` and `services/event-bus/` are deprecated (M8), then removed (M9).
3. **Subscriber semantics move client-side.** api-gateway's Subscribe→REST bridge re-implements the server-side glob + workflow_id scope filter + terminal-event stream-close in-process against a JetStream durable consumer, ending the REST watch on the terminal event exactly as today. This is the hardest migration and is called out as such.
4. **Transport security reuses the cert-manager identity — no second secret system.** The JetStream link gains **TLS with cert-manager-issued client certificates**, and NATS is configured with `verify_and_map` so each service authenticates as a cert-mapped NATS user — per-service identity stays under the single ADR-020 mechanism (no net-new nkeys/creds to distribute, no ADR-020 Option-C shared-secret regression). Topic authorization keys off that cert-mapped identity via NATS subject permissions. Migrating a caller to the direct client is **not "done" until it dials over TLS with its cert-manager identity.** Closing the pre-existing plaintext gap (§Context) is in scope for this epic, not deferred; because cert-manager already provisions these per-service certs in the umbrella chart, it adds **no manual secret step** to the kind first-run (ADR-041 — the EPIC #1370 zero-secret quickstart is preserved).
5. **Python is additive, in the optional SDK.** The `agents/sdk` gains a `nats-py`-based eventing path re-implementing the same conventions. Nothing breaks (no Python eventing exists). This is **consistent with ADR-013** — the SDK stays optional and adapters remain gRPC-only; it lifts ADR-022's own "no NATS client in agents" pre-commitment, and scopes the NATS client to the **SDK eventing path only** (capability dispatch stays gRPC).
6. **Deprecate then remove, per repo convention.** M8: shared client lands, all four call sites migrate, facade marked deprecated (kept running). M9: `services/event-bus/`, `event_bus.proto`, and the gRPC stubs are hard-removed once no caller references them.

## Rationale

**ADR-022's four pillars, answered:**

| ADR-022 pillar | Disposition under this ADR |
|----------------|----------------------------|
| Stable versioned contract (gRPC type system) | AsyncAPI (`zynax-events.yaml` + `zynaxschemarev`) is the real contract; the shared library gives every Go caller compile-time types. **Honest gap:** Python loses gRPC-generated types → mitigated by golden-event byte-compat tests + `zynaxschemarev`, not eliminated. |
| Transport security (mTLS) | Relocated to TLS on the JetStream link with **cert-manager-issued client certs** cert-mapped to per-service NATS users (Decision #4) — the same ADR-020 identity mechanism, not a second secret system. The facade's mTLS only ever covered inbound gRPC, not the plaintext NATS link — so this **closes a real gap** rather than losing one. |
| Policy chokepoint (topic authz, namespace isolation, rate limits) | The facade enforces none today (§Context). Topic/namespace authorization moves to **NATS subject permissions** keyed off the cert-mapped service identity (a JetStream-native primitive, ADR-040 §1). Rate limits split: **edge** request-rate is an api-gateway concern (planned sibling ADR-044, #1574, M8.F — co-accepted in this M8 batch), while **internal publish-rate** abuse (a compromised producer flooding a subject) is bounded natively by **JetStream per-stream limits** (max msgs/bytes/consumers) — a control the facade never had either, so no regression. Nothing enforced is dropped. |
| ADR-001/ADR-013 compliance | A narrow ADR-001 exception (Decision #1) — eventing is fire-and-forget pub/sub, not inter-service RPC. **No ADR-013 exception is needed**: the SDK stays optional and adapters stay gRPC-only (Decision #5); the `nats-py` path lifts ADR-022's own "no NATS client in agents" pre-commitment, not an ADR-013 rule. ADR-008's "durability in JetStream" is strengthened, not weakened. |

| Option | Assessment |
|--------|------------|
| **Retire the facade; direct JetStream via `libs/zynaxevents` + AsyncAPI contract; deprecate M8 / remove M9** | ✅ **Chosen** — deletes a service that adds a network hop and zero policy; keeps every convention as versioned library code; aligns with ADR-040 §1 (delegate to JetStream, don't wrap it); makes the pre-existing NATS-plaintext gap a tracked deliverable. |
| Keep the facade, finally build the M7 chokepoint in it | ✗ Rejected — builds a bespoke Zynax service to do what NATS account/subject permissions already do (ADR-040 §1 anti-pattern), and keeps the per-publish hop. |
| Hybrid: thin control-plane service + direct NATS for hot paths (ADR-022 Option 3) | ✗ Rejected — two code paths and a service that still enforces nothing; worst of both. |
| Direct NATS with conventions copy-pasted per service (no shared module) | ✗ Rejected — the #1149 disjoint-filter and DLQ rules would drift per copy; the `libs/` precedent (ADR-006 `go.work` monorepo, `libs/zynaxconfig` / `libs/zynaxobs`) keeps shared conventions in one versioned module. |

## Consequences

- **Positive:**
  - One fewer service to build, deploy, monitor, and scale — eliminating **both** of ADR-022's stated costs (the extra service *and* the per-`Publish` network hop).
  - Conventions become a single versioned `libs/zynaxevents` module — one place to fix the #1149 disjoint-filter / DLQ / terminal-close logic, consumed like `libs/zynaxobs`.
  - Durability stays entirely in JetStream (ADR-008 upheld); the AsyncAPI contract is unchanged for external consumers.
  - Forces the **pre-existing** NATS plaintext/no-auth gap closed (cert-manager client certs + link TLS) as a tracked deliverable rather than a latent risk hidden behind facade mTLS.
- **Negative / trade-off (load-bearing):**
  - **Every publishing/subscribing service now dials the broker directly** — a wider blast-radius surface than a single facade identity. There is no new credential *type* (each service reuses its existing cert-manager cert, Decision #4), but every service can now reach NATS directly; mitigated by least-privilege per-service subject permissions, though the surface is real and new.
  - **Python consumers need a client path.** The optional SDK must carry `nats-py` and re-implement the conventions; a second implementation of the depth-4/DLQ/glob rules must be kept byte-compatible with the Go one (golden-event tests). It lives in the optional SDK, so it does not change ADR-013 (adapters stay gRPC-only).
  - **Contract-drift risk without the gRPC type system.** With no `EventBusService` stub, nothing structurally forces a producer's payload to match a consumer's expectation at compile time across languages; the guardrail is AsyncAPI + `zynaxschemarev` + golden-event tests, which is weaker than generated types.
  - The api-gateway Subscribe→REST bridge must faithfully reproduce server-side glob + workflow_id-scope + terminal-close **client-side**; a subtle miss leaks a never-closing REST watch. Covered by BDD, but it is the riskiest single migration.
- **Neutral / follow-up required:**
  - `.feature` contract + golden-event byte-compat + DLQ/durable-consumer BDD committed before implementation (ADR-016/019).
  - Migration order: land `libs/zynaxevents` → migrate the three publishers → migrate the api-gateway subscriber → provision NATS link TLS + cert-mapped users → mark facade deprecated (M8) → remove `services/event-bus/`, `event_bus.proto`, stubs (M9).
  - Runtime smoke before claiming done: exercise workflow-lifecycle publish and the `WatchWorkflowLogs` stream end-to-end (twice — persistence/terminal-close bites the second run), confirming the terminal event still closes the watch. CI-green alone is insufficient.
  - **kind first-run impact (ADR-041).** Since ADR-041 makes kind the sole local runtime that mirrors production, the mandated NATS link-TLS provisioning now runs on **every laptop first-run**, not just in prod. Because it reuses cert-manager (already installed by the umbrella chart) rather than a net-new nkeys secret, `make demo` / `zynax up` auto-issues the certs with **no manual secret step** — the one-command, zero-secret EPIC #1370 quickstart is preserved.
  - The `nats-py` client is scoped to the SDK eventing path only; capability dispatch stays gRPC and must not acquire a NATS client (consistent with ADR-013 — no exception carved).
