# ADR-045: Admission-policy delegation — engine allow-list to a K8s-native `ValidatingAdmissionPolicy`; the concurrent-invocation quota stays custom

**Status:** Accepted  **Date:** 2026-07-03  **Corrected:** 2026-07-06 (see note below)

> **Correction (2026-07-06, M8.G delivery — #1575):** this ADR was authored alongside ADR-043,
> before its `Workflow` CR landed. M8.E shipped the engine hint as the CR **spec field
> `spec.engine`** (lifted into the submit path's `EngineHint` by the reconciler), not as the
> `zynax.io/engine-hint` annotation on `metadata` this ADR assumed. The VAP therefore validates
> **`object.spec.engine`** — the sentences below are corrected accordingly. The compiler-side
> REST guard (`checkRoutingPolicy`) still reads the `zynax.io/engine-hint` manifest annotation;
> that path is unchanged. Canvas: `docs/spdd/1575-admission-policy/canvas.md`.
**Related:** ADR-040 (**amends** — extends the §1 delegation list: the coarse engine allow-list moves to a Kubernetes-native admission policy, but the concurrent-invocation quota is explicitly kept custom, honouring §6, which keeps the AI scheduling/intelligence core as protected custom code), ADR-043 (**depends on** — the forthcoming thin `Workflow` CRD front-end foreshadowed by ADR-040 §3, authored alongside this cluster and not yet in the register; Kubernetes admission only intercepts API-server objects, so the delegated gate can attach *only* to that `Workflow` CR — it is inert for the REST path until the CR lands), ADR-039 (CRD-native control-plane precedent — the `Agent` CRD as the reconciled source of truth), ADR-011 (admission may enforce *structural* validation that `spec/schemas/` owns, never scheduling semantics), ADR-034 (routing identity/`ManifestWorkflowID` unaffected). Implements EPIC #1575 (M8.G); a REASONS Canvas + story issues are created when the M8 `feat:` work begins (canvas planned at `docs/spdd/1575-admission-policy/canvas.md`).

## Context

`services/workflow-compiler/internal/domain/policy_gate.go` runs two checks inside `Server.CompileWorkflow` (`services/workflow-compiler/internal/api/server.go`), first violation wins:

1. **Engine allow-list** (`checkRoutingPolicy`) — pure string set-membership: reject when the `zynax.io/engine-hint` annotation is absent from a namespace's non-empty `AllowedEngines`. No Zynax intelligence; the engine-*fit* decisioning (capability↔engine matching, hint semantics) lives elsewhere in the compiler (kindrouter / IR compile), **not** in this file.
2. **Concurrent-invocation quota** (`checkCapabilityQuota`) — queries an injected `ActiveInvocationCounter.ActiveCount(ns)` and rejects when `active >= MaxConcurrent`.

Both are **fail-open**. Config is env-var-only and single-namespace (`ZYNAX_POLICY_*` in `services/workflow-compiler/internal/config/config.go`; M6 limitation). Critically, `buildPolicyGate()` in `services/workflow-compiler/cmd/workflow-compiler/main.go` constructs the gate with `counter = nil`, so **in production only the allow-list is live; the compiler quota is contract-only** (exercised solely by `policy_gate_test.go` with a stub). A second quota gate exists at dispatch time — `services/engine-adapter/internal/infrastructure/quota_check.go` (`QuotaChecker`, same fail-open semantics) — but it is **also contract-only today**: `NewQuotaChecker` is constructed only in `quota_check_test.go`, `QuotaChecker.Check` has no caller in any production path (activities / engine / `cmd`), and there is **no concrete `ActiveInvocationCounter` implementation anywhere in the repo** (only a test `stubCounter`). So *neither* quota gate actually enforces anything in production: the compiler's is wired into `main.go` but starved (`counter = nil`), and the engine-adapter's is never wired at all. #1575 requires this ADR to decide the fate of *both*.

The decisive constraint: **the whole path is REST → gRPC, not the Kubernetes API.** `zynax apply` (`cmd/zynax/cmd/apply.go`) POSTs to the gateway, `ApplyService.ApplyWorkflow` (`services/api-gateway/internal/domain/apply.go`) calls the compiler over gRPC, and the gate fires inside the compiler. **No Kubernetes object is ever created for a `kind: Workflow`** (ADR-040 §Context: it compiles on-the-fly to IR). Kyverno, OPA-Gatekeeper, and `ValidatingAdmissionPolicy` intercept *only* API-server CREATE/UPDATE requests — they cannot see a REST submission. There is no admission machinery in the repo today (grep finds it only in docs; the sole real CRD is `helm/zynax-agent-registry/crds/agents.zynax.io.yaml`). The only future admission attach point is the thin `Workflow` CR of the forthcoming ADR-043 (M8.E) — not yet in the register and not built yet. This ADR is therefore **coupled to ADR-043** and must say plainly what guards the REST path until then.

## Decision

**We will delegate the coarse engine allow-list to a Kubernetes-native `ValidatingAdmissionPolicy` bound to the ADR-043 `Workflow` CR, keep the compiler-side allow-list as the guard for the REST path until that path is retired, and keep the concurrent-invocation quota custom — it does not map faithfully to any admission or `ResourceQuota` primitive.**

1. **Allow-list → CEL `ValidatingAdmissionPolicy` (VAP).** When a `Workflow` CR is applied to the API server (ADR-043 / GitOps), a VAP validates `object.spec.engine in params.allowedEngines` (unset/empty `spec.engine` = platform default engine, always admitted), with per-namespace `AllowedEngines` carried in a bound `ValidatingAdmissionPolicyBinding`/params object. VAP is chosen because it is **beta and enabled by default on kind's `kindest/node:v1.29.2`** (`scripts/e2e/cluster-up.sh`) — **no new controller dependency**, satisfying ADR-040 §1 ("use a Kubernetes primitive, not a new Zynax service"). Kyverno is the recorded fallback (see Rationale). GA `admissionregistration.k8s.io/v1` requires bumping the node image to ≥1.30.
2. **Concurrent-invocation quota stays custom, consolidated onto the dispatch-time `QuotaChecker` in engine-adapter** — the only layer that *can* observe live in-flight counts (admission and `ResourceQuota` cannot). We do **not** attempt a `ResourceQuota` mapping: `ResourceQuota` counts K8s *objects*, admission is a point-in-time gate on CREATE, and neither can see Zynax runtime invocation counts. **Correctness precondition (load-bearing): the engine-adapter `QuotaChecker` is itself contract-only today** — `NewQuotaChecker` is never constructed in production, `Check` has no caller, and no concrete `ActiveInvocationCounter` exists (only a test stub). Consolidating onto it is therefore only sound *after* it is wired live (see Consequences); until then this ADR treats the quota as **unenforced on both gates**, not as a working gate we are converging on. The compiler's starved quota check (`counter = nil`) is removed from `policy_gate.go`, but its BDD contract **does not migrate** to the engine-adapter gate until that gate is actually invoked — migrating it sooner would leave a green contract test passing against a never-called component and advertise quota protection that does not exist (an ADR-020 zero-trust / DoS regression).
3. **Compiler allow-list retained for REST until REST is retired.** Because admission cannot see REST submissions, `checkRoutingPolicy` stays live in the compiler for the `zynax apply`/gateway path. Enforcement converges on VAP **only** for CRs applied via `kubectl`/GitOps; when ADR-043's CR becomes the sole authoring surface (or the REST path is routed through CR creation), the compiler allow-list is deleted and VAP is the single source of enforcement. This ADR does not retire REST; it defines the interim as dual-guard.
4. **Structural-only scope.** The VAP enforces set-membership on a CR spec field (`spec.engine`) — a structural check aligned with `spec/schemas/`. It **must not** absorb engine-fit / scheduling semantics, which remain protected custom core in the compiler (ADR-040 §6). ADR-040 §5 defers "*Admission/conversion webhooks for CRDs … until a `v1beta1` exists*"; a `ValidatingAdmissionPolicy` is **in-tree CEL, not an admission webhook**, so it sits outside that deferral and is in scope now.

## Rationale

| Option | Assessment |
|--------|------------|
| **CEL `ValidatingAdmissionPolicy` for the allow-list + keep the dispatch-time quota custom** | ✅ **Chosen** — no new controller (VAP is built-in beta on v1.29), delegates the one truly-generic check (string set-membership) to the platform, and is honest that the quota needs live runtime state admission cannot provide. Enforcement is GitOps-diffable and RBAC-scoped. |
| Kyverno (or OPA-Gatekeeper) for the allow-list | ✗ Rejected (fallback) — a full policy engine + webhook is a new cluster dependency for one set-membership rule; reserved only if we later need policy features VAP/CEL cannot express (multi-resource lookups, generation, mutation). Recorded so the fallback is on the record. |
| Map the concurrent quota to `ResourceQuota` | ✗ Rejected — `ResourceQuota` counts K8s objects, not in-flight Zynax invocations; a `Workflow` CR persists after its run completes, so object count ≠ live concurrency. Would silently change what "quota" means. |
| Admission-time quota (VAP/Kyverno counting existing CRs) | ✗ Rejected — point-in-time on CREATE, blind to running invocations and to the fail-open semantics both current gates rely on; admission is fail-closed by default. The engine-adapter re-check would still be required, so this adds a second, weaker gate for no gain. |
| Delegate now, before ADR-043 lands | ✗ Rejected — with no `Workflow` CR there is no attach point; delegating today would *drop* allow-list enforcement for every `zynax apply`. Hence the explicit ADR-043 dependency and the interim compiler retention. |
| Move engine-fit decisioning into admission too | ✗ Rejected — that is the protected Zynax core (ADR-040 §6); it does not live in `policy_gate.go` and must not migrate to CEL. |

## Consequences

- **Positive:** The one generic check (engine allow-list on a CR) becomes a standard, GitOps-diffable, RBAC-scoped Kubernetes policy with zero new runtime dependency on v1.29 kind. The quota's *design intent* is consolidated onto the one layer that can hold live in-flight counts (dispatch-time `QuotaChecker`), removing the dead `counter = nil` compiler quota and the confusing appearance of two competing quota gates. This is consolidation of intent, **not** of working enforcement — see the required follow-up: neither quota gate runs today, so the payoff is realised only once `QuotaChecker` is wired. Aligns M8.G with ADR-040's delegation principle without violating §6.
- **Negative / trade-off (load-bearing):** Enforcement is **split by entry path until REST is retired** — CRs are guarded by VAP, `zynax apply` by the retained compiler allow-list — so the same rule lives in two engines with a drift risk (the interim dual-guard). Full delegation is **blocked on the forthcoming ADR-043**; if that CR path slips, M8.G delivers **nothing enforceable for the allow-list** and — because the quota gate is also un-wired — only a quota *touch*, not enforcement. **Hard sequencing:** this ADR MUST NOT retire or delete the compiler-side `checkRoutingPolicy`, and the VAP + `ValidatingAdmissionPolicyBinding` MUST NOT ship, until ADR-043 is Accepted and its `Workflow` CR — carrying the engine hint as `spec.engine` (corrected 2026-07-06; landed in M8.E) — has landed; otherwise the allow-list loses its only live guard. VAP semantics are **fail-closed** where the compiler gate was fail-open, a behaviour change to document. Staying on VAP *beta* pins a semantics dependency on the v1.29 node image; GA needs a ≥1.30 bump.
- **Neutral / follow-up required:** ADR-043's `Workflow` CR landed (M8.E) carrying the engine hint as **`spec.engine`** — the field CEL reads (corrected 2026-07-06; the annotation-on-`metadata` wording this ADR originally used never shipped). Add the `ValidatingAdmissionPolicy` + `ValidatingAdmissionPolicyBinding` (params carrying per-namespace `AllowedEngines`) under `helm/`; wire an e2e case on kind. **Wire `QuotaChecker` into the pre-dispatch path before any consolidation:** construct it in engine-adapter wire-up/`main`, inject a real `ActiveInvocationCounter` (backed by the task-broker) in place of the test stub, and call `Check` before `DispatchCapabilityActivity`. **Only then** delete the compiler quota path and migrate its BDD contract to the engine-adapter gate — doing so earlier would green-light a contract test against a never-called component (a false-enforcement guarantee). Supersede the original single-gate design in `docs/spdd/768-policy-enforcement/canvas.md`. Multi-namespace `AllowedEngines` (today one namespace via env vars) is naturally expressed as multiple bindings — revisit when multi-ns policy config lands.
