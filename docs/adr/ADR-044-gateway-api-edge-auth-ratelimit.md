# ADR-044: Edge delegation — gateway bearer-auth + rate-limit to a Gateway API edge (Envoy Gateway)

**Status:** Accepted  **Date:** 2026-07-03
**Related:** ADR-040 (**amends** — adds edge L7 bearer-auth + rate-limiting to the Decision §1 delegated list, and refines the §Consequences "must not be over-applied" caution: an external L7 edge policy is a generic primitive, not the protected domain core), ADR-020 (cert-manager mTLS stays the internal-mesh boundary — untouched by this ADR), ADR-022 (relates — the EventBus gRPC-facade chokepoint for topic-authz/namespace-isolation/rate-limits is a *different* boundary and is **not** moved to the edge here), ADR-041 (kind is the unified runtime — the edge must install and run locally in one command), ADR-039 (relates — HPA scale-out is precisely why per-pod in-process limiting is wrong). Resolves EPIC #1574 (M8.F); canvas `docs/spdd/1574-gateway-api-edge-auth-ratelimit/canvas.md`.

---

## Context

Two external-edge concerns are re-implemented **in-process** inside api-gateway today:

- **Bearer auth** — `services/api-gateway/internal/api/auth.go` (26 LOC): a constant-time compare of `"Bearer "+key` against the `Authorization` header. The key is `ZYNAX_GW_API_KEY`; `services/api-gateway/cmd/api-gateway/main.go` refuses to start on an empty key (unless `ZYNAX_GW_DEV_INSECURE=1`).
- **Rate limiting** — `services/api-gateway/internal/api/ratelimit.go` (89 LOC): a **per-pod, in-memory** per-IP token bucket (`golang.org/x/time/rate`, `RATE_LIMIT_RPS`/`RATE_LIMIT_BURST`), tested by `ratelimit_test.go` (218 LOC). Both are wired **per-route** in `handler.go` `RegisterRoutes` (the two mutating POSTs get `rl.Middleware(requireBearer(...))`; `DELETE` gets auth only; GET endpoints are open).

The in-memory limiter is the footgun EPIC #1574 targets: under HPA with **N** replicas each holds an independent bucket, so the effective limit is **N × RPS** — there is no global limit. The auth check is a shared-secret bearer token, which ADR-020 §Rejected Alternatives (Option C) itself calls a *stepping stone*.

ADR-040 §Decision 1 already delegates **auth transport** (cert-manager mTLS + RBAC) and **networking** (NetworkPolicy) to Kubernetes — but its delegated list does **not** include edge L7 bearer-auth or rate-limiting, and its §Consequences warns the thin-Zynax principle "must not be over-applied to strip genuine domain logic." So #1574 poses an ADR-gated, potentially-Rejected question: *is an external HTTP edge policy (bearer-auth + a global rate limit) a generic K8s-delegatable primitive, or protected Zynax core?*

The relevant facts on kind: `helm/zynax-api-gateway/templates/service.yaml` exposes ClusterIP/NodePort (host 8080 → nodePort 30080, the first-run contract asserted in `scripts/e2e/cluster-up.sh`); the HTTP edge (:8080) is **plaintext** (api-gateway `tlsSecretName` defaults to `""` — cert-manager mTLS covers only backend gRPC hops); and **no Gateway API or HTTPRoute object exists, and nothing fronts the api-gateway edge** — greenfield (the only Ingress in the repo is the optional Uptrace login-UI Ingress at `helm/charts/uptrace/templates/ingress.yaml`, gated on `.Values.ingress.enabled`). The `zynax-gw-api-key` Secret is created by `scripts/e2e/cluster-up.sh` and mounted as a Deployment env; the CLI defines `--api-key` (`cmd/zynax/cmd/root.go`), passed to the client (`cmd/zynax/client/gateway.go`) which sends it as `Authorization: Bearer <key>`.

## Decision

**We will delegate edge bearer-token authentication and rate-limiting from in-process api-gateway middleware to a Kubernetes Gateway API edge (Envoy Gateway). api-gateway keeps only its Zynax core — `DetectKind` routing and compile/submit fan-out.**

1. **Gateway API + Envoy Gateway as the edge.** Gateway API (`gateway.networking.k8s.io`) is the standard successor to Ingress. We pick **Envoy Gateway** as the reference implementation because it is a CNCF project with Gateway API conformance, it ships a `SecurityPolicy` (API-key / JWT bearer validation) **and** a **global** rate-limit path (`BackendTrafficPolicy` → the Envoy global rate-limit service), and it installs on **kind** as a single Helm release. `GatewayClass` / `Gateway` / `HTTPRoute` plus `SecurityPolicy` / `BackendTrafficPolicy` objects are added to the umbrella chart (new templates under `helm/`).

2. **Global rate-limit, not per-pod (load-bearing).** The edge enforces **one shared limit across all N api-gateway replicas** via Envoy's global rate-limit service (a small rate-limit backend + its store), fixing the HPA-multiplication bug. This is the primary reason edge-global beats per-pod in-process: an in-memory bucket cannot coordinate across replicas, so it is structurally incapable of a correct global limit.

2a. **First-run stays light (ADR-041 / #1370): the global limiter is profile-gated.** The global rate-limit backend (an Envoy rate-limit service **plus its store, e.g. Redis**) is **not** required in the minimal #1370 zero-secret quickstart. A single-replica first-run has no N×RPS multiplication to correct, so the quickstart profile ships **edge auth only** (`SecurityPolicy`) and the rate-limit backend is **opt-in**, enabled by a Helm value in the HPA / multi-replica (CI/prod) profile where scale-out makes a coordinated limit meaningful. This honors ADR-041's "clone → one command → result" floor explicitly instead of silently raising it, while still delivering the correct global limit exactly where it matters.

3. **Bearer auth at the edge; the CLI Bearer UX is the target, verified — not assumed.** An Envoy Gateway `SecurityPolicy` with **API-Key auth** extracts a header value and does an **exact match** against a referenced Secret whose data is a `client-name → key` mapping. Two mechanics must be reconciled, and they are **acceptance items rather than assertions**: (a) extracting from the `Authorization` header yields the full `Bearer <key>` string (prefix included), so either the stored key material carries the `Bearer ` prefix or the policy is configured to strip it — the CLI header construction (`cmd/zynax/client/gateway.go`) must **not** have to change; and (b) today's `zynax-gw-api-key` Secret is a single `api-key` literal, whereas the edge expects the client-name→key shape, so the Secret is re-shaped (or a dedicated edge Secret is added) as part of this work rather than consumed as-is. The exact `SecurityPolicy` type, header/prefix handling, and Secret data shape are pinned in the M8.F Canvas and proven by an acceptance test that the unchanged `zynax --api-key` path still authenticates end-to-end.

4. **Delete, do not bypass.** `services/api-gateway/internal/api/auth.go`, `ratelimit.go`, `ratelimit_test.go`, the per-route `RegisterRoutes` wrapper wiring in `handler.go`, and the api-key plumbing in `services/api-gateway/cmd/api-gateway/main.go` (`validateConfig`, `cfg.APIKey`, the Deployment secret env in `helm/zynax-api-gateway/templates/deployment.yaml`) are **removed**. api-gateway retains `DetectKind` routing, compile/submit fan-out, dry-run, namespace propagation, SSE `/logs`, `/outputs`, cancel, probes, `/metrics`, and OTEL middleware.

4a. **The deleted auth is replaced, not merely removed (defense-in-depth, ADR-020).** ADR-020 §Context relied on api-gateway bearer auth to protect the external boundary. Today the api-gateway `NetworkPolicy` (`helm/zynax-api-gateway/templates/networkpolicy.yaml`) admits `:8080` from **any pod in the cluster** — its ingress rule carries **no `from:` selector**. So once §4 removes the in-process check, any in-cluster workload (or a direct `kubectl port-forward` to the Service — the maintainer's documented access pattern) could submit/cancel workflows **unauthenticated, bypassing the Envoy edge entirely**. To prevent this net loss of defense-in-depth, this ADR **adds a `from:` selector** (pod/namespace selector for the Envoy Gateway pods, plus the kubelet probe path) to that NetworkPolicy ingress rule, restricting `:8080` to the Gateway so it becomes the **sole** ingress to the api-gateway Service. The in-process bearer check is thus swapped for an **edge-plus-NetworkPolicy** control, not simply deleted.

5. **One-command local is preserved (ADR-041), with explicit install ordering.** A single umbrella Helm release renders and validates all manifests up front and **cannot** guarantee that the `Gateway` / `HTTPRoute` / `SecurityPolicy` / `BackendTrafficPolicy` custom resources are admitted only after their CRDs are registered and the Envoy Gateway controller is reconciling — a real first-run failure mode on a fresh kind cluster. So the edge installs as an **ordered prerequisite phase, not a co-installed subchart**: (i) apply the Gateway API CRDs **and** the Envoy Gateway CRDs (`SecurityPolicy` / `BackendTrafficPolicy`) and controller, and **wait for the controller to be `Ready`**; then (ii) apply the Zynax `Gateway` / `HTTPRoute` / `SecurityPolicy` / `BackendTrafficPolicy` objects. This ordering is an explicit step in `scripts/e2e/cluster-up.sh` (and the equivalent Argo **sync-wave** on the GitOps path), invoked by `zynax up` and `make demo`, so kind first-run stays **one command**. The Gateway listener fronts the api-gateway Service; the kind host:8080 first-run contract is re-pointed to the Gateway listener (NodePort/port-forward now targets the edge).

6. **mTLS is unchanged (ADR-020) — and does not cover this hop.** cert-manager mTLS remains the **internal-mesh** boundary for inter-service **gRPC**; per ADR-020 the api-gateway HTTP `:8080` edge is **plaintext and outside the mTLS mesh**. So deleting in-process bearer auth (§4) would otherwise leave that HTTP endpoint with **neither** bearer auth **nor** mTLS — which is exactly why §4a locks the Service down to the Gateway as its sole ingress. The Gateway is the **external** boundary; edge TLS termination is possible but out of scope for this ADR.

7. **The ADR-022 chokepoint is a different boundary.** ADR-022 §3 locates EventBus topic-authz / namespace-isolation / rate-limits at the **gRPC facade** (inter-service publish/subscribe). That is not this boundary and is **not** moved to the HTTP edge; it stays owned by ADR-022 (M8.H may revisit it separately).

## Rationale

| Option | Assessment |
|--------|------------|
| **Gateway API edge (Envoy Gateway) — global rate-limit + `SecurityPolicy` bearer auth** | ✅ **Chosen** — a *true* global limit across HPA replicas; standard, diffable, GitOps-able Gateway API objects; CNCF-idiomatic; installs on kind in one Helm release; shrinks api-gateway to its Zynax core. |
| Keep in-process middleware (status quo) | ✗ Rejected — the per-pod in-memory bucket multiplies the effective limit to **N × RPS** under HPA; re-implements an L7 primitive the ecosystem does better; the exact footgun #1574 names. |
| ingress-nginx annotations | ✗ Rejected — not Gateway-API-native; its rate-limit annotations are per-controller-replica and approximate, not a coordinated global limit; drifts from the Gateway API direction. |
| Istio ingress / full service mesh | ✗ Rejected — heavyweight for a single external edge; a large local-dev footprint on kind; overshoots the need. |
| A bespoke Zynax edge-authz service | ✗ Rejected — ADR-040 §1 explicitly rejects building a Zynax service for a delegatable primitive. |
| Reject the epic / defer to M9 | ✗ Deferred-then-rejected — a Rejected ADR was a valid outcome, but it leaves the HPA-multiplication bug and keeps edge L7 in-process against ADR-040's direction; the maintainer accepted delegation. |

## Consequences

- **Positive:** The rate limit becomes **correct under HPA** (one global limit, not N buckets). Roughly ~333 LOC of bespoke edge middleware and its tests are deleted, not bypassed. The edge is now standard Gateway API objects — diffable, `kubectl`-able, Argo-syncable — strengthening the M8 CNCF posture. api-gateway shrinks to the Zynax-owned core (kind-routing + compile/submit fan-out).
- **Negative / trade-off (load-bearing):** A **new infra dependency (the Envoy Gateway control plane + proxy) in every profile including kind first-run**, which raises the resource floor and cold-start and adds an edge that must be healthy for *any* request to land (the global rate-limit backend and its store are **profile-gated** per §2a, so the minimal #1370 first-run stays lighter). **Local-dev complexity rises**: operators now reason about and debug `Gateway`/`HTTPRoute`/`SecurityPolicy`/`BackendTrafficPolicy` objects, and the host:8080 NodePort/port-forward contract retargets to the Gateway listener. The **api-key flow migrates** from a Deployment `secretKeyRef` to an edge `SecurityPolicy` reference — `scripts/e2e/cluster-up.sh` secret creation and its read-back consumers (e2e, `scripts/demo/kind-demo.sh`, bench) must be re-pointed. **Security regression under ADR-020 unless compensated:** deleting the in-process bearer check leaves the plaintext, non-mTLS `:8080` edge protected only by the Envoy edge — but the api-gateway NetworkPolicy today admits `:8080` from any in-cluster pod, so this ADR **must** ship the §4a `from:`-selector lockdown making Envoy the sole ingress; without it, any in-cluster workload or a direct `kubectl port-forward` to the Service could submit/cancel workflows unauthenticated. The edge also becomes a **new single locus for auth failures**.
- **Neutral / follow-up required:** Create the M8.F REASONS Canvas + story issues. Add the Gateway/HTTPRoute/SecurityPolicy/BackendTrafficPolicy templates under `helm/`, and install the Gateway API + Envoy Gateway CRDs/controller as an **ordered prerequisite phase** (CRDs + controller `Ready` **before** the CRs, per §5) wired into `scripts/e2e/cluster-up.sh` / `zynax up` / `make demo` and the Argo sync-wave; retarget the kind host:8080 contract to the Gateway; add the §4a NetworkPolicy `from:`-selector lockdown. Decide the **GET-endpoints posture** (stay open vs. now authenticated at the edge) in the Canvas. Acceptance per #1574: unauthenticated requests rejected at the Envoy edge; the api-gateway NetworkPolicy admits `:8080` **only from the Envoy Gateway** (no unauthenticated in-cluster bypass); the `SecurityPolicy` type + `Bearer `-prefix handling + Secret shape proven so the **unchanged** `zynax --api-key` path authenticates end-to-end; `auth.go`/`ratelimit.go` **deleted not bypassed**; the global limit **demonstrated under HPA scale-out**; e2e green on **both** engine legs. The ADR-022 EventBus chokepoint stays owned by ADR-022.
